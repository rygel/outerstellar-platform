How far should the DaisyUI migration go?

Your project has ~90 custom CSS classes, 60+ templates, and 817 lines of hand-written CSS. DaisyUI can replace most of it.

A

Full replacement (Recommended)

Rip out all 817 lines of custom CSS. Replace every custom class with DaisyUI semantic equivalents (panel → card, button-secondary → btn btn-ghost, chip → badge, data-table → DaisyUI table, etc.). Delete input.css entirely. Your templates become cleaner and your theme system becomes DaisyUI's built-in 30+ themes.

B

Incremental — components first, layout later

Replace component classes (buttons, panels, badges, tables, forms, modals) with DaisyUI. Keep layout classes (shell, sidebar, topbar, content, footer) as custom CSS. Faster to ship, less risky, but you'll maintain a hybrid CSS file.

C

Drop-in only — add DaisyUI, replace nothing

Install DaisyUI alongside existing CSS. Only use DaisyUI for new components going forward. Old templates keep their custom classes. Zero migration risk, but you now have two competing design systems.
