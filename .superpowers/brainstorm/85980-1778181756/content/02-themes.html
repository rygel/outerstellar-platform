How should the theme system work with DaisyUI?

You currently have a custom theme system with 32 themes defined in themes.json, injected via CSS variables at runtime. DaisyUI has 30+ built-in themes with its own color semantics. These can coexist in different ways.

A

Use DaisyUI themes exclusively (Recommended)

Replace your 32 custom themes with DaisyUI's built-in themes. Set theme via data-theme="dark" on <html> (you already have this attribute). DaisyUI handles all color semantics automatically — no CSS variables needed. Users pick from DaisyUI's theme list. Delete themes.json and the Kotlin theme engine.

B

Custom DaisyUI themes from your existing palettes

Convert your 32 themes from themes.json into DaisyUI custom theme definitions (CSS @theme blocks). Keeps your exact brand colors but gets DaisyUI's component integration. More upfront work, but your users keep the themes they know.

C

Hybrid — DaisyUI themes + CSS variable overrides

Use DaisyUI's theme system but keep your custom themes.json as overrides on top. DaisyUI sets base colors, your theme engine adjusts specific variables. Maximum flexibility but you maintain two systems.
