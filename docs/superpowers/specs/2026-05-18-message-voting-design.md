# Message Voting System Design

**Date:** 2026-05-18
**Status:** Approved
**PR target:** `develop`

## Overview

Add upvote/downvote on messages. Public net score visible to all, individual votes private. Core feature with REST API + HTMX web UI.

## Scope

- Messages only (not contacts or other entities)
- Upvote (+1) and downvote (-1) per user per message
- Toggle: voting same direction again removes the vote; voting opposite direction flips it
- Public score display, private vote identities

## Schema

Flyway migration `V10__add_message_votes.sql`:

```sql
CREATE TABLE plt_message_votes (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    message_sync_id VARCHAR(36)  NOT NULL,
    user_id         UUID         NOT NULL,
    direction       SMALLINT     NOT NULL CHECK (direction IN (1, -1)),
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    FOREIGN KEY (message_sync_id) REFERENCES plt_messages(sync_id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES plt_users(id) ON DELETE CASCADE
);
CREATE UNIQUE INDEX ux_plt_message_votes_message_user ON plt_message_votes(message_sync_id, user_id);
CREATE INDEX idx_plt_message_votes_message_sync_id ON plt_message_votes(message_sync_id);
CREATE INDEX idx_plt_message_votes_user_id ON plt_message_votes(user_id);
```

Unique constraint on `(message_sync_id, user_id)` enforces one vote per user per message. `ON DELETE CASCADE` cleans up votes when messages or users are deleted.

## Data Model

`platform-core/src/main/kotlin/io/github/rygel/outerstellar/platform/model/VoteModels.kt`:

```kotlin
data class MessageVote(
    val id: Long = 0,
    val messageSyncId: String,
    val userId: UUID,
    val direction: Int,
    val createdAt: Instant,
)

data class VoteScore(
    val messageSyncId: String,
    val upvotes: Int,
    val downvotes: Int,
    val netScore: Int,
    val userVote: Int?,
)
```

- `direction` is always +1 or -1
- `VoteScore.userVote` is `null` if the requesting user hasn't voted, or +1/-1 if they have

## Repository

`platform-core/src/main/kotlin/io/github/rygel/outerstellar/platform/persistence/VoteRepository.kt`:

```kotlin
interface VoteRepository {
    fun save(vote: MessageVote)
    fun update(userId: UUID, messageSyncId: String, direction: Int)
    fun delete(userId: UUID, messageSyncId: String)
    fun findByUserAndMessage(userId: UUID, messageSyncId: String): MessageVote?
    fun countByMessage(messageSyncId: String): Pair<Int, Int>  // (upvotes, downvotes)
    fun findScoresByMessages(messageSyncIds: List<String>, userId: UUID?): List<VoteScore>
    fun findScoreByMessage(messageSyncId: String, userId: UUID?): VoteScore
}
```

Implementations: `JooqVoteRepository` (platform-persistence-jooq) and `JdbiVoteRepository` (platform-persistence-jdbi).

`findScoresByMessages` returns scores for multiple messages in one query, used by the message list page to avoid N+1 queries.

## Service

`platform-core/src/main/kotlin/io/github/rygel/outerstellar/platform/service/VoteService.kt`:

Constructor: `VoteRepository`, `MessageRepository` (for existence check), `AuditRepository?`, `EventPublisher`.

Methods:

| Method | Description |
|--------|-------------|
| `vote(messageSyncId, userId, direction)` | If no existing vote: create. If same direction: remove (toggle off). If different: update to new direction. Validates message exists. Audit log. |
| `removeVote(messageSyncId, userId)` | Delete user's vote. Audit log. |
| `getScores(messageSyncIds, userId?)` | Batch fetch scores. Returns map of syncId -> VoteScore. |
| `getScore(messageSyncId, userId?)` | Single message score. |

### Vote logic (state machine)

| Existing vote | New action | Result |
|--------------|-----------|--------|
| None | Upvote | Create +1 |
| None | Downvote | Create -1 |
| +1 | Upvote | Delete (toggle off) |
| -1 | Downvote | Delete (toggle off) |
| +1 | Downvote | Update to -1 |
| -1 | Upvote | Update to +1 |

## API Routes

`platform-web/src/main/kotlin/io/github/rygel/outerstellar/platform/web/VoteApi.kt` implementing `ServerRoutes`.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/v1/messages/{syncId}/vote` | Required | Cast or toggle vote. Body: `{"direction": 1}` or `{"direction": -1}` |
| DELETE | `/api/v1/messages/{syncId}/vote` | Required | Remove user's vote |
| GET | `/api/v1/messages/{syncId}/vote` | Optional | Get score (includes user's vote if authenticated) |

Responses:
- POST: 200 with `VoteScore` JSON on success, 404 if message not found
- DELETE: 204 on success, 404 if no vote to remove
- GET: 200 with `VoteScore` JSON

## Web UI (HTMX)

### Component route

Add to `ComponentRoutes.kt` or a new `VoteComponentRoutes.kt`:

| Method | Path | Description |
|--------|------|-------------|
| GET | `/components/messages/{syncId}/vote` | Returns vote buttons + score as HTML fragment |

This is the HTMX target — voting buttons POST via HTMX and swap this fragment.

### Message list integration

The message list page (rendered by `HomePageFactory` via JTE templates) includes vote buttons for each message. The vote fragment is rendered inline during initial page load. HTMX `hx-post` and `hx-swap="outerHTML"` on the vote buttons targets the component route.

### Vote fragment HTML structure

Each message's vote section:
- Upvote arrow (highlighted if user upvoted)
- Net score display
- Downvote arrow (highlighted if user downvoted)

HTMX attributes on each button:
```
hx-post="/components/messages/{syncId}/vote"
hx-vals='{"direction": 1}'
hx-target="closest .vote-container"
hx-swap="outerHTML"
```

The component route handles both rendering (GET) and voting (POST), returning HTML fragments. The JSON API routes (`/api/v1/messages/{syncId}/vote`) are separate and serve API consumers.

## DI Wiring

1. `VoteRepository` registered in `WebModule.kt` as `single<VoteRepository> { JooqVoteRepository(get()) }` (or JDBI variant)
2. `VoteService` registered as `single { VoteService(get(), get(), getOrNull(), get()) }`
3. `VoteApi` registered in `buildApiRoutes()` in `App.kt`
4. `VoteService` passed to `HomePageFactory` and `HomeRoutes` for message list rendering
5. `VoteService` passed to `ComponentRoutes` for the HTMX fragment endpoint

## Non-goals

- Voting on contacts or other entities (could be added later with polymorphic target)
- Reply threads or nested voting
- Vote-based sorting or ranking algorithm
- Vote analytics or trending
- Anonymous voting or vote weighting
- Desktop client voting (sync support) — deferred to a future iteration
